Java 5 → Java 8
15 Hands-On Exercises

Rewrite each Java 5 snippet in modern Java 8 idiom. Attempt every task before opening the solution — the struggle is where the learning happens.

A. Lambdas (1–4) B. Streams (5–12) C. Optional (13) D. Date/Time (14) E. Capstone (15) Interview Questions

A. Lambdas & Functional Interfaces

Goal: stop writing anonymous inner classes. Learn the shapes: Predicate, Function, Consumer, Supplier, and method references.

01

Comparator: sort orders by amount

WARM-UP
Java 5 code
Collections.sort(orders, new Comparator<Order>() {
    public int compare(Order a, Order b) {
        return Double.compare(a.getAmount(), b.getAmount());
    }
});

Task: rewrite as (a) a lambda, then (b) using Comparator.comparing. Bonus: sort descending, then by customer name as tie-breaker.

Solution
// (a) lambda
orders.sort((a, b) -> Double.compare(a.getAmount(), b.getAmount()));

// (b) comparing — preferred, reads like English
orders.sort(Comparator.comparingDouble(Order::getAmount));

// bonus: descending + tie-breaker
orders.sort(Comparator.comparingDouble(Order::getAmount).reversed()
        .thenComparing(Order::getCustomerName));

Why it matters: Comparator.comparing / thenComparing chains are the idiomatic answer; writing the two-arg lambda manually is already considered old-fashioned.

02

Runnable: start a worker thread

WARM-UP
Java 5 code
Thread t = new Thread(new Runnable() {
    public void run() {
        System.out.println("Processing order queue...");
    }
});
t.start();

Task: rewrite with a lambda. Then answer: why does this work for Runnable but not for an interface with two abstract methods?

Solution
Thread t = new Thread(() -> System.out.println("Processing order queue..."));
t.start();

Why: a lambda can only implement a functional interface — exactly one abstract method. That's what the compiler targets. @FunctionalInterface documents and enforces this.

03

Reusable filter logic with Predicate

CORE
Java 5 code
List<Order> result = new ArrayList<Order>();
for (Order o : orders) {
    if (o.getAmount() > 1000 && "DE".equals(o.getCountry())) {
        result.add(o);
    }
}

Task: define two separate Predicate<Order> variables (high-value, German) and combine them with .and(). Filter the list using them (loop is fine here — streams come next).

Solution
Predicate<Order> highValue = o -> o.getAmount() > 1000;
Predicate<Order> german    = o -> "DE".equals(o.getCountry());

Predicate<Order> wanted = highValue.and(german);   // also: .or(), .negate()

List<Order> result = new ArrayList<>();
for (Order o : orders) {
    if (wanted.test(o)) result.add(o);
}

Why: behavior becomes a value you can name, pass, combine, and test in isolation. This mental shift is the whole point of Java 8.

04

Method references — all four kinds

CORE

Task: rewrite each lambda as a method reference and name which of the four kinds it is:

s -> s.toUpperCase()
s -> Integer.parseInt(s)
x -> System.out.println(x)
() -> new ArrayList<String>()
Solution
String::toUpperCase     // 1. instance method of an arbitrary object of a type
Integer::parseInt       // 2. static method
System.out::println     // 3. instance method of a particular object
ArrayList::new          // 4. constructor reference

Rule of thumb: if the lambda does nothing but call one existing method with the same arguments, use the reference — it reads better and interviewers expect you to know the four kinds.

B. Streams API

The biggest daily-life change since Java 5. Pattern: source → intermediate ops (lazy) → terminal op. Assume class Order { String id; String customer; String status; String country; double amount; List<String> items; }.

05

Filter + transform + collect

WARM-UP
Java 5 code
List<String> ids = new ArrayList<String>();
for (Order o : orders) {
    if ("ACTIVE".equals(o.getStatus())) {
        ids.add(o.getId());
    }
}
Collections.sort(ids);

Task: one stream pipeline.

Solution
List<String> ids = orders.stream()
        .filter(o -> "ACTIVE".equals(o.getStatus()))
        .map(Order::getId)
        .sorted()
        .collect(Collectors.toList());   // Java 16+: .toList()

Note: the pipeline does nothing until the terminal operation (collect) runs — intermediate ops are lazy.

06

Counting and matching

WARM-UP
Java 5 code
int count = 0;
boolean anyBig = false;
for (Order o : orders) {
    if ("DE".equals(o.getCountry())) count++;
    if (o.getAmount() > 10000) anyBig = true;
}

Task: replace with count() and anyMatch(). Also look up allMatch / noneMatch.

Solution
long count = orders.stream()
        .filter(o -> "DE".equals(o.getCountry()))
        .count();

boolean anyBig = orders.stream()
        .anyMatch(o -> o.getAmount() > 10000);

Why: anyMatch short-circuits — it stops at the first hit, unlike the Java 5 loop as written.

07

Find first — meet Optional

CORE
Java 5 code
Order found = null;
for (Order o : orders) {
    if (o.getId().equals(searchId)) { found = o; break; }
}
if (found != null) {
    System.out.println(found.getCustomer());
} else {
    System.out.println("not found");
}

Task: use findFirst() and handle both cases without any null or if.

Solution
orders.stream()
      .filter(o -> o.getId().equals(searchId))
      .findFirst()
      .map(Order::getCustomer)
      .ifPresentOrElse(
          System.out::println,                    // Java 9+
          () -> System.out.println("not found"));

// pure Java 8 alternative:
String name = orders.stream()
      .filter(o -> o.getId().equals(searchId))
      .findFirst()
      .map(Order::getCustomer)
      .orElse("not found");

Why: findFirst() returns Optional<Order> — the stream API forces you into null-safe style.

08

Sum and average with primitive streams

CORE
Java 5 code
double total = 0;
for (Order o : orders) total += o.getAmount();
double avg = orders.isEmpty() ? 0 : total / orders.size();

Task: use mapToDouble. Get sum, average, max in one pass with summaryStatistics().

Solution
double total = orders.stream().mapToDouble(Order::getAmount).sum();

DoubleSummaryStatistics stats = orders.stream()
        .mapToDouble(Order::getAmount)
        .summaryStatistics();
// stats.getSum(), stats.getAverage(), stats.getMax(), stats.getCount()

Why: IntStream / LongStream / DoubleStream avoid boxing. Know mapToInt, boxed(), and IntStream.range(0, n) as the modern for-loop.

09

groupingBy — the map-of-lists killer

CORE
Java 5 code
Map<String, List<Order>> byStatus = new HashMap<String, List<Order>>();
for (Order o : orders) {
    List<Order> list = byStatus.get(o.getStatus());
    if (list == null) {
        list = new ArrayList<Order>();
        byStatus.put(o.getStatus(), list);
    }
    list.add(o);
}

Task: one line with Collectors.groupingBy. Then: count per status, and total amount per status (downstream collectors).

Solution
Map<String, List<Order>> byStatus = orders.stream()
        .collect(Collectors.groupingBy(Order::getStatus));

Map<String, Long> countPerStatus = orders.stream()
        .collect(Collectors.groupingBy(Order::getStatus, Collectors.counting()));

Map<String, Double> totalPerStatus = orders.stream()
        .collect(Collectors.groupingBy(Order::getStatus,
                 Collectors.summingDouble(Order::getAmount)));

Why: groupingBy + downstream collectors is the single most asked stream question in interviews. Also learn partitioningBy (boolean split) and toMap.

10

Joining strings

WARM-UP
Java 5 code
StringBuilder sb = new StringBuilder();
for (int i = 0; i < orders.size(); i++) {
    if (i > 0) sb.append(", ");
    sb.append(orders.get(i).getId());
}
String csv = sb.toString();

Task: use Collectors.joining with a delimiter, prefix "[" and suffix "]".

Solution
String csv = orders.stream()
        .map(Order::getId)
        .collect(Collectors.joining(", ", "[", "]"));   // → [A1, B2, C3]
11

flatMap — flatten nested collections

STRETCH
Java 5 code
Set<String> allItems = new TreeSet<String>();
for (Order o : orders) {
    for (String item : o.getItems()) {
        allItems.add(item);
    }
}

Task: one pipeline producing a sorted distinct list of all item names across all orders. Explain in one sentence when you need flatMap instead of map.

Solution
List<String> allItems = orders.stream()
        .flatMap(o -> o.getItems().stream())
        .distinct()
        .sorted()
        .collect(Collectors.toList());

The sentence: use flatMap when the mapping function returns a stream/collection per element and you want one flat stream instead of a stream of streams — map would give Stream<Stream<String>>.

12

Top-N: sort, limit, skip

CORE

Task: from orders, produce the customer names of the 3 highest-value ACTIVE orders, highest first. No Java 5 version given — write it directly from the requirement.

Solution
List<String> top3 = orders.stream()
        .filter(o -> "ACTIVE".equals(o.getStatus()))
        .sorted(Comparator.comparingDouble(Order::getAmount).reversed())
        .limit(3)
        .map(Order::getCustomer)
        .collect(Collectors.toList());

Watch out: order of operations matters — map before sorted would lose access to the amount. Interviewers love asking you to reorder a broken pipeline.

C. Optional

13

Kill the null-check pyramid

CORE
Java 5 code
String city = "UNKNOWN";
if (order != null) {
    Customer c = order.getCustomer();
    if (c != null) {
        Address a = c.getAddress();
        if (a != null && a.getCity() != null) {
            city = a.getCity().toUpperCase();
        }
    }
}

Task: one Optional chain. Then answer: why should Optional not be used for fields or method parameters?

Solution
String city = Optional.ofNullable(order)
        .map(Order::getCustomer)
        .map(Customer::getAddress)
        .map(Address::getCity)
        .map(String::toUpperCase)
        .orElse("UNKNOWN");

Why not fields/params: Optional is designed as a return type to make "maybe absent" explicit at API boundaries. As a field it adds allocation and isn't serializable; as a parameter it forces callers into awkward wrapping — overloads or null-object patterns are cleaner. Also never call .get() without checking; prefer orElse / orElseGet / orElseThrow / map / filter.

D. java.time

14

Replace Date/Calendar

CORE
Java 5 code
Calendar cal = Calendar.getInstance();
cal.add(Calendar.DAY_OF_MONTH, 30);
Date dueDate = cal.getTime();
SimpleDateFormat fmt = new SimpleDateFormat("dd.MM.yyyy");
String formatted = fmt.format(dueDate);   // fmt is NOT thread-safe!

Task: rewrite with java.time. Also compute the number of days between two dates, and the same instant in the Europe/Berlin and Asia/Kolkata time zones.

Solution
LocalDate dueDate = LocalDate.now().plusDays(30);
String formatted = dueDate.format(DateTimeFormatter.ofPattern("dd.MM.yyyy"));

long days = ChronoUnit.DAYS.between(orderDate, dueDate);

ZonedDateTime berlin  = ZonedDateTime.now(ZoneId.of("Europe/Berlin"));
ZonedDateTime kolkata = berlin.withZoneSameInstant(ZoneId.of("Asia/Kolkata"));

Why: everything in java.time is immutable and thread-safe — every plusDays returns a new object. Know the cast of characters: LocalDate, LocalDateTime, Instant (machine time), ZonedDateTime, Duration, Period.

E. Capstone

15

Order report — everything together

STRETCH

Task: given List<Order>, produce a report string like:

ACTIVE: 12 orders, total 45,230.50 EUR (top: Müller GmbH)
SHIPPED: 8 orders, total 12,100.00 EUR (top: Schmidt AG)

Requirements: group by status; per group compute count, total amount, and the customer of the highest-value order; sort groups by total descending; join lines with \n. Use only streams — no loops, no mutable state.

Solution
String report = orders.stream()
    .collect(Collectors.groupingBy(Order::getStatus))
    .entrySet().stream()
    .map(e -> {
        List<Order> group = e.getValue();
        double total = group.stream().mapToDouble(Order::getAmount).sum();
        String top = group.stream()
                .max(Comparator.comparingDouble(Order::getAmount))
                .map(Order::getCustomer)
                .orElse("-");
        return new AbstractMap.SimpleEntry<>(total,
            String.format("%s: %d orders, total %,.2f EUR (top: %s)",
                          e.getKey(), group.size(), total, top));
    })
    .sorted(Map.Entry.<Double,String>comparingByKey().reversed())
    .map(Map.Entry::getValue)
    .collect(Collectors.joining("\n"));

If you solved this unaided, Phase 1 is done. Rewriting it later with records (as the pair type) and toList() makes a nice Phase 3 revisit.

Interview Questions From This Material

These are the questions German enterprises and consultancies actually ask at the Java 8 level, roughly in order of frequency.

1. What is a functional interface? What does @FunctionalInterface do?VERY COMMON

Interface with exactly one abstract method (SAM). The annotation makes the compiler enforce it. Default and static methods don't count against the limit. Name the big four: Function, Predicate, Consumer, Supplier — and BiFunction/UnaryOperator variants.

2. Lambda vs. anonymous inner class — differences?VERY COMMON

Key points: lambdas don't create a separate .class file per use (invokedynamic), this inside a lambda refers to the enclosing instance (not the lambda), lambdas only work for functional interfaces, and captured local variables must be effectively final in both.

3. Intermediate vs. terminal operations; what does "lazy" mean?VERY COMMON

Intermediate ops (filter, map, sorted) return a stream and do nothing on their own; a terminal op (collect, forEach, count, reduce) triggers the whole pipeline in one pass. Follow-up trap: a stream can be consumed only once — reusing it throws IllegalStateException.

4. map vs. flatMap?VERY COMMON

map transforms 1→1; flatMap transforms 1→stream and flattens. Classic example: list of orders → all items (exercise 11). Also appears with Optional: Optional.flatMap for methods that themselves return Optional.

5. How does groupingBy work? What are downstream collectors?COMMON

Be ready to write exercise 9 on a whiteboard: groupingBy(classifier), then with counting(), summingDouble(), mapping(). Mention partitioningBy for boolean splits and toMap (plus its merge-function parameter for duplicate keys — a favorite follow-up).

6. Optional best practices — when should you NOT use it?COMMON

Return types: yes. Fields, method parameters, collections of Optionals: no. Never .get() without isPresent; prefer orElse/orElseGet/orElseThrow. Bonus point: orElse always evaluates its argument, orElseGet is lazy — a very popular gotcha question.

7. What is "effectively final" and why do lambdas require it?COMMON

A local variable that's never reassigned after initialization. Lambdas capture values, not variables; allowing mutation would create race conditions and stale-value confusion since the lambda may run later or on another thread.

8. parallelStream() — when is it a good idea?COMMON

Rarely by default: it uses the shared ForkJoinPool, helps only for large, CPU-bound, stateless, cheap-to-split workloads, and hurts for I/O-bound work or small collections. Saying "I'd measure first and avoid it in request-handling code" is the senior answer.

9. Default methods — why were they added, and what about the diamond problem?COMMON

Added so interfaces (like Collection) could evolve without breaking implementors — that's how forEach/stream were retrofitted. If two interfaces provide the same default method, the class must override and can delegate via InterfaceName.super.method().

10. What changed in HashMap internals in Java 8?SENIOR / TRIVIA

Buckets convert from linked lists to red-black trees past a threshold (8), turning worst-case O(n) lookups into O(log n) under heavy collisions. Also: PermGen was replaced by Metaspace in the JVM — both are classic German-enterprise interview trivia.

11. Why is java.time better than Date/Calendar?OCCASIONAL

Immutable and thread-safe (SimpleDateFormat famously isn't), sane API (months not zero-indexed), clear separation of machine time (Instant) vs. human time (LocalDate/ZonedDateTime), proper time-zone and DST handling.

12. Whiteboard task you should expectPRACTICAL

"Given a list of employees, return the average salary per department, sorted descending" — or some variant. It's exercises 8 + 9 + 12 combined. If you can write it fluently without an IDE, you're ready for this round.
